以下の記事では、Unityでの開発における基本的な命名規則や、ファイル構成の考え方について紹介します。初心者の方にとっては、コードやプロジェクト構造を見やすく整理するための第一歩となるポイントです。是非参考にしてみてください。

# クラス名

**ポイント**：クラス名は**PascalCase**（単語の頭文字を大文字、続く文字は小文字）の形式で命名します。  
**例**：`PlayerController`、`GameManager`、`EnemyAI` など

**理由**：  
- PascalCaseはC#言語の標準的な慣習であり、UnityもC#ベースの開発環境であるため、これに従うことで読みやすさが向上します。  
- クラスは基本的に"設計図"や"概念"を表すので、単語の頭文字を大文字にすることで、ほかの要素（変数や関数）との区別がしやすくなります。

# 変数名

**ポイント**：変数名は**camelCase**（先頭は小文字、以降の単語の頭文字を大文字）で命名します。  
**例** : `playerHealth`、`spawnPosition`、`currentScore`

**補足**
- 変数はオブジェクトやデータの状態を示すので、"何を表しているのか"が分かる名前を付けましょう。  
- `hp`や`pos`のような短い省略形は、プロジェクト内で統一されているか、特に有名な略称でない限り避けると可読性が向上します。  
- ブール型（true/false）変数には `is` や `can`、`has` などのプレフィックス（接頭辞）を付けるとより意味が明確になります。  
**例**：`isJumping`、`canAttack`、`hasKey`

:::details 注意
変数名の先頭が小文字でも、Unityエディタでは自動的に大文字で表示される仕様になっています。そのため、特に気にする必要はありません。

# 関数名

**ポイント**：関数名もクラス名同様に**PascalCase**で命名します。ただし、"動詞"や"処理内容を示すワード"を優先して使うことで、動作が直感的に分かるようにします。  
**例**：`StartGame()`、`UpdateScore()`、`MovePlayer()`、`SpawnEnemy()`

**理由**：  
- 関数は"何をする処理なのか"を示すべきなので、`Get`、`Update`、`Initialize`、`Reset`といった動詞や行動に関連する単語を積極的に使用すると、コードを読む際に処理内容がすぐに把握できます。

# ゲームオブジェクトの命名

Unityエディタ上で扱う**ゲームオブジェクト（GameObject）**や**プレハブ（Prefab）**にも、一貫した命名ルールを用いるとチーム開発や後でのメンテナンスがしやすくなります。

**おすすめ**：  
- オブジェクト名もPascalCaseで統一すると見やすいです。  
- 複数の類似オブジェクト（例えば敵キャラクター）がある場合は、`Enemy_Goblin`、`Enemy_Skeleton` のように、接頭辞として`Enemy_`を付けるなど、カテゴリーが分かるようにすると整理しやすくなります。  
- プレハブは特定のフォルダ(`Prefabs`)にまとめ、`Player.prefab`、`MainCamera.prefab`のように機能が明確な名前を付けておくと便利です。

# ファイル構成について

Unityプロジェクトは、`Assets`フォルダの中にスクリプト、プレハブ、素材（画像、音声、モデル）を格納するのが基本です。プロジェクトが大きくなるとファイルが散らばり、どこに何があるか分からなくなりやすいので、早い段階からルールを決めておくと良いでしょう。

**一般的なフォルダ構成例**：  
```
Assets/
    Scripts/
        Managers/
            GameManager.cs
            UIManager.cs
        Player/
            PlayerController.cs
            PlayerHealth.cs
        Enemies/
            EnemyAI.cs
    Prefabs/
        Player.prefab
        Enemy_Goblin.prefab
        Enemy_Skeleton.prefab
    Scenes/
        MainMenu.unity
        Level1.unity
    Materials/
        PlayerMaterial.mat
        EnemyMaterial.mat
    Textures/
        UI/
            Button.png
            Icon.png
        Environment/
            Grass.png
            Stone.png
    Audio/
        BGM/
            MainTheme.mp3
        SFX/
            Jump.wav
            Explosion.wav
```

**ポイント**：  
- 機能や役割ごとにフォルダを分ける。  
- `Scripts`フォルダには、さらにサブフォルダを作って、システムごと（Managers、Player、Enemiesなど）にまとめます。  
- `Prefabs`、`Materials`、`Textures`、`Audio`など、アセットの種類ごとに整理します。  
- シーン（Scene）も、`Scenes`フォルダを用意してそこに格納しましょう。

**メリット**：  
- 「この機能を実装しているスクリプトはどこにある？」と探す手間が減ります。  
- チーム開発でも、新規メンバーが参加した時に「Scriptsの中は機能別のフォルダに分かれていますよ」と伝えるだけで、おおよその場所を把握しやすくなります。

## まとめ

- **クラス名**：PascalCaseで、クラスは概念を表す。  
- **変数名**：camelCaseで、意味が明確な名前を付ける（ブールは`is～`で始めるなど工夫）。  
- **関数名**：PascalCaseで、動作を表す動詞を含める。  
- **ゲームオブジェクト**：PascalCaseで命名し、カテゴリ別に接頭辞を付けるなど工夫する。  
- **ファイル構成**：プロジェクトをフォルダで機能別、種類別に整理し、探しやすくする。
