## はじめに

CLAUDE.mdに「日本語で回答」と書いてあるのに、Claudeが英語で返してくる。昨日まで守ってくれていた指示が突然無視される。

原因は、**CLAUDE.mdが339行まで膨張していたこと**でした。Anthropic公式ベストプラクティスには「短い指示の方が良い」と明記されていたのです。

https://code.claude.com/docs/en/best-practices

この記事では、1年半のCLAUDE.md運用で経験した7つの失敗パターンと解決策を共有します。

:::message
**核心**: 指示が増えるほど、特定の指示だけでなく**全体の遵守率が均一に低下します**。339行のCLAUDE.mdでは、1つ1つの指示が50-60%程度でしか守られなくなっていました（体感）。
:::

## 7つの失敗パターン

### パターン1: 肥大化で遵守率が崩壊

21行→339行→42行の変遷。追記を繰り返し339行になった結果、「日本語で回答」すら守られなくなりました。42行に圧縮したら遵守率が回復。

### パターン2: 同じルールを3箇所に書いて混乱

「日本語で回答」をCommunication、Response Format、Execution Detailsの3箇所に微妙に異なる表現で書いた結果、日によって挙動が変わりました。**Single Source of Truth -- 同じルールは1箇所だけ**が鉄則です。

### パターン3: Mermaid図を義務化したら全部見えなくなった

ターミナルでBash経由出力は折りたたまれる仕様のため、「何も見えない報告」が量産されました。**ルール追加前に実環境で検証すべし**。

:::message alert
対策: デフォルトはASCII図解（`text`ブロック形式）にし、Bash経由の図表生成は禁止。
:::

### パターン4: 「禁止」だけ伝えて行き詰まった

「Bashでファイル操作するな」と禁止事項だけ列挙した結果、Claudeが何を使えばいいか分からず停止。**「〜するな、代わりに〜しろ」をセットで書く**ことで解決。

### パターン5: タスクに無関係な指示がスキップされた

最も衝撃的な失敗。Claude Codeには**タスクとの関連性が低い指示をスキップする仕組み**がある可能性が指摘されています。

https://github.com/anthropics/claude-code/issues/15443

対策として特に効果的だったのは**WHY（理由）の併記**です。

```markdown
- IMPORTANT: コミットメッセージは必ず日本語で書く
  - WHY: チーム全員が日本語話者のため、英語コミットは可読性を下げる
```

:::message
WHYを書くことで「このルールは現在のタスクに高度に関連している」とClaudeが判断し、遵守率が体感30%→95%に向上しました。
:::

### パターン6: スキル一覧ベタ書きでトークン浪費

30個以上のスキルを直接列挙して80-100行に。MCPサーバーのツール定義だけで数万トークンを消費する事例も報告されています。

https://deploystack.io/blog/how-mcp-servers-use-your-context-window

**段階的開示（Progressive Disclosure）**で3行に圧縮し、約98%のトークン削減を実現。

```markdown
## リソース参照
- カタログ: @~/.claude/README.md
- スキル: ~/.claude/skills/*/SKILL.md
- エージェント: ~/.claude/agents/*.md
```

| 項目 | Before | After |
|------|--------|-------|
| CLAUDE.md行数 | 80-100行 | 3行 |
| 推定トークン | 2,000-3,000 | 約50 |

### パターン7: /clearせずに応答5分超

長時間セッションで会話履歴が蓄積し、応答が5分超に。月末の請求を見て驚愕しました。**タスク切り替え時の/clear**と**`/usage`コマンドでの定期監視**で劇的に改善。不要なMCPサーバーの無効化も効果的です。

## CLAUDE.md設計の3原則

### 原則1: 42行ルール

CLAUDE.mdは42行（20-30個の指示）に抑え、詳細は段階的に外部化する。

| レベル | ファイル | 読み込みタイミング |
|-------|---------|------------------|
| Level 1 | CLAUDE.md（42行） | 全タスクで自動 |
| Level 2 | README.md（カタログ） | @参照で必要時 |
| Level 3 | SKILL.md / agents/*.md | Task tool実行時のみ |

### 原則2: 禁止+代替のセット記述

否定形の指示には必ず代替案とWHY（理由）をセットで記述。

### 原則3: 追加前テスト

新ルールは仮追加→実タスクで3回検証→全成功なら正式化。

## まとめ

| 教訓 | 要点 |
|------|------|
| 指示は150個以内に | LLM認知限界 |
| 重複指示は矛盾を生む | Single Source of Truth |
| ルール追加前に検証 | 3回テスト原則 |
| 禁止には代替案をセット | ポジティブフレーミング |
| タスク無関係な指示はスキップされる | WHY併記で対策 |
| 詳細は段階的開示で外部化 | Progressive Disclosure |
| 定期的な/clear | コンテキスト汚染防止 |

**339行で失敗した僕だから言える。CLAUDE.mdは短いほうが強い。**
