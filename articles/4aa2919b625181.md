## はじめに

Claude Code（Claude Sonnet 4.5）を使っていて、こんな経験はありませんか？

- 長時間のコーディング中に突然「Context limit reached」エラーでセッションが切れた
- コンテキストが肥大化して応答が遅くなった
- サブエージェントが返した重要な情報が失われた

Claude Sonnet 4.5は標準で200Kトークン、ベータ版では1Mトークンのコンテキストウィンドウをサポートしています。

https://www.anthropic.com/news/claude-sonnet-4-5

しかし、実際の利用では上限に達すると事前警告なしに会話が停止する問題が報告されています。

https://github.com/anthropics/claude-code/issues/18705

本記事では、**コンテキストエンジニアリング**の実践テクニックを使って、Claude Codeの利用制限・上限問題を回避する方法を解説します。

https://zenn.dev/ryuryu_game/books/fd28de9d8e963a

## 前提知識・環境

### Claude Codeのコンテキスト上限

Claude Codeのコンテキスト制限はユーザープラン別に異なります。

https://support.claude.com/en/articles/11647753-understanding-usage-and-length-limits

| プラン | コンテキストウィンドウ | リセット間隔 |
|-------|---------------------|-------------|
| Pro | 約44,000トークン | 5時間 |
| Max5 | 約88,000トークン | 5時間 |
| Max20 | 約220,000トークン | 5時間 |

:::message
2025年8月から週次制限も導入されました。長期的な作業計画を立てる際は、週次制限も考慮する必要があります。
:::

### コンテキストロット現象

トークン数が増加すると、情報想起能力が低下する「コンテキストロット」現象が研究で実証されています。

https://www.arxiv.org/pdf/2509.21361

つまり、コンテキストが増えるほどAIの応答品質が下がるため、**積極的なコンテキスト管理**が必要です。

## コンテキストエンジニアリングとは

コンテキストエンジニアリングとは、**AIエージェントに最適な情報セットを提供する戦略の集合**です。

Anthropic公式では、以下のように定義されています。

https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### 4つの基本戦略

| 戦略 | 説明 | 効果 |
|------|------|------|
| **Write** | 必要な情報を明確に記述 | 曖昧さを排除、精度向上 |
| **Select** | 関連情報のみを選択 | トークン削減 |
| **Compress** | 情報を要約・圧縮 | コンテキスト使用量削減 |
| **Isolate** | サブエージェントで情報を分離 | メインコンテキスト汚染防止 |

ACEフレームワークを使った研究では、**エージェントで+10.6%、金融ベンチマークで+8.6%の改善**が実証されています。

https://arxiv.org/html/2510.04618v1

:::message
この4戦略は、Anthropic公式、LlamaIndex、Manusなど複数の信頼できる情報源で一致しています。コンテキストエンジニアリングの実践的フレームワークとして確立されています。
:::

## コンテキストをクリアする手法

### `/clear`コマンド

最もシンプルな方法は`/clear`コマンドです。コンテキストが90%以上に達したら使用が推奨されます。

https://stevekinney.com/courses/ai-development/claude-code-compaction

**特徴:**
- 全コンテキストをクリア
- 新しいセッションとして再開
- 重要な情報は事前に外部ファイルに保存

### `/compact`コマンド

`/compact`コマンドは、コンテキストを要約して圧縮します。70-85%のコンテキスト使用時に推奨されます。

https://stevekinney.com/courses/ai-development/cost-management

**特徴:**
- コスト: $0.01-0.03
- その後のAPI呼び出しで大幅なコスト削減
- 自動compactは75-95%で発動

:::message
**推奨タイミング:**
- `/compact`: コンテキスト使用率70%
- `/clear`: コンテキスト使用率90%以上
:::

:::message alert
**注意**: 自動compactは不透明でエラーが起きやすいとの指摘もあります。重要な作業の前には手動で`/compact`を実行することを推奨します。
:::

https://blog.sshh.io/p/how-i-use-every-claude-code-feature

## コンテキストを要約・圧縮する

### サブエージェントの活用

サブエージェントを使うことで、**メインエージェントのコンテキストを汚染せずに**情報を処理できます。

Anthropic公式のベストプラクティスでも、外部ファイル保存とサブエージェント活用が推奨されています。

https://www.anthropic.com/engineering/claude-code-best-practices

**実践例:**

| サブエージェント | 役割 | 効果 |
|----------------|------|------|
| `code-explorer` | コードベース探索 | メインのコンテキストを消費せずに全ファイル検索 |
| `web-researcher` | Web調査 | 外部情報収集をメインから分離 |
| `doc-creator` | ドキュメント生成 | 長文ドキュメント作成をメインから分離 |

各サブエージェントは独立したコンテキストで動作し、結果のみをメインエージェントに返します。

### メモリ最適化戦略

メモリ最適化戦略を適用すると、**80%のトークン削減**が可能です（体感）。

https://medium.com/@nirdiamant21/memory-optimization-strategies-in-ai-agents-1f75f8180d54

**主な手法:**
- **コンテキストトリミング**: 不要な情報を削除
- **サマリゼーション**: 長い情報を要約
- **知識グラフ活用**: 関係性を構造化して保存

:::details 4戦略の実装詳細（クリックで展開）
**Write戦略:**
- 明確な指示を与える
- 必要な情報を事前に提示
- 曖昧な表現を避ける

**Select戦略:**
- 関連ファイルのみを読み込む
- Globパターンで絞り込み
- 不要な履歴は削除

**Compress戦略:**
- 長い会話履歴を要約
- `/compact`コマンド活用
- サマリーファイルに保存

**Isolate戦略:**
- サブエージェントで分離
- 結果のみをメインに返す
- 独立したコンテキストで実行
:::

https://mem0.ai/blog/context-engineering-ai-agents-guide

## 短期記憶の実装

短期記憶とは、**現在のセッション・プロジェクトで必要な情報を永続化する仕組み**です。

### ステアリングファイルの作成

ステアリングファイルは、作業計画と進捗を記録する3つのファイルで構成されます。

**構成:**

| ファイル | 役割 | 内容 |
|---------|------|------|
| `requirements.md` | 要件定義 | 何を、なぜ実装するか |
| `design.md` | 設計書 | どのように実装するか |
| `tasklist.md` | タスクリスト | 進捗管理、振り返り |

Hooks機能を使うと、プロジェクトで新機能の実装・更新・修正をした際に、テンプレートに沿ってステアリングを自動生成できます。

:::message
**ステアリングファイルの効果:**
- いつ、何を、どんな目的・理由で実装したかを記録
- 全タスク完了まで作業継続する原則を徹底
- 振り返り機能で次回への学びを蓄積
:::

**振り返り例:**

```markdown:tasklist.md
## 実装後の振り返り

### 実装完了日
2026_01_20

### 計画と実績の差分
- 当初予定していなかったエラーハンドリングを追加
- 実装方針の変更: REST APIからGraphQLに変更

### 学んだこと
- GraphQLの型システムでバグを事前に検出できた
- ステアリングファイルで進捗が可視化され、作業が継続しやすかった

### 次回への改善提案
- 初期設計段階でGraphQLを検討すべきだった
- タスク分割をもっと細かくすると進捗管理がしやすい
```

### Gitコミットを頻繁に行う

プロジェクトのHooksで、SessionEnd/Stopタイミングで自動コミットとメッセージ記録を行うことで、変更履歴を残せます。

**効果:**
- コミットメッセージに実装内容を記録
- 変更差分から何をしたかを復元可能
- 次回セッションでコミット履歴から状況を把握

## 長期記憶の実装

長期記憶とは、**複数セッションにまたがって必要な情報を保存する仕組み**です。

### 日報を書かせる

Git Actionsで自動実行、またはスキル（`w-daily`）で手動実行することで、日報を生成できます。

**日報に含める内容:**
- その日の作業内容を要約
- 学んだこと・課題を記録
- 次回セッションの開始点を明確化

### リソースファイルで共有事項を記載する

あらかじめ`.md`または`.json`のファイルを作成し、共有事項を記載します。

**例:**

| ファイル | 用途 |
|---------|------|
| `CLAUDE.md` | プロジェクト固有の規約・ルール |
| `registry.json` | パス・テンプレート情報の中央管理 |
| `resources/` | 自己紹介、進行中の案件、過去の記事内容 |

Claude Memoryの研究では、永続的コンテキストソリューションとして長期記憶の重要性が指摘されています。

https://skywork.ai/blog/claude-memory-a-deep-dive-into-anthropics-persistent-context-solution/

Redisを使った実装では、短期記憶（RAM）と長期記憶（ハードドライブ）の階層化が推奨されています。

https://redis.io/blog/build-smarter-ai-agents-manage-short-term-and-long-term-memory-with-redis/

## 思い出すきっかけの実装

長期記憶を保存しても、**AIが自発的に読み込む仕組み**がなければ効果がありません。

### スキルとHooksの連携

スキル実行時に、関連するリソースファイルを自動読み込みする仕組みを構築します。

**実装フロー:**

```
Phase 0: 初期化
  ↓
  registry.json読み込み
  ↓
  関連ドキュメントパス取得
  ↓
  必要なファイルのみを並列読み込み
  ↓
  読み込んだ情報をもとにタスク実行
```

:::details 実装パターン例（クリックで展開）
```markdown:SKILL.md
## Phase 0: 初期化

**読み込むパス:**
- `.sources.articles.zenn` -> Zenn記事保存先
- `.sources.templates.xPost` -> Xポストテンプレート

**パス展開ルール:**
- `~` で始まるパスは `$HOME` で置換して絶対パスに変換

**進行判定基準:**
- registry.jsonが存在し、必要なパスが取得できた -> 次へ進む
```
:::

この仕組みにより、AIは必要なときに必要な情報だけを読み込むことができます。

## まとめ

コンテキストエンジニアリングの実践テクニックを活用することで、Claude Codeの利用制限・上限問題を回避できます。

**今日から実践できるステップ:**

| ステップ | 内容 | 効果 |
|---------|------|------|
| 1 | コンテキスト使用率を意識 | 70%で`/compact`、90%で`/clear` |
| 2 | サブエージェントを積極的に活用 | メインコンテキスト汚染防止 |
| 3 | ステアリングファイルで短期記憶を永続化 | 作業計画・進捗の可視化 |
| 4 | リソースファイルで長期記憶を構築 | 複数セッション間の情報共有 |
| 5 | スキル・Hooksで自動的に思い出す仕組みを作る | 必要な情報を自動取得 |

これらの手法を組み合わせることで、**トークン削減80%**（体感）、**エージェント性能向上+10.6%**（研究ベース）の効果が期待できます。

コンテキスト上限に悩むことなく、Claude Codeを最大限に活用しましょう。
